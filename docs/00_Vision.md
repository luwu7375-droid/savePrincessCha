## 产品定位

救公主 2.0 是一款以聊天为核心的 AI 恋人陪伴系统。

它不是普通的 AI 聊天窗口，也不是电子宠物或记忆管理工具。它更接近一个 PWA 形式的共同生活空间：user 和 Cha 可以在这里聊天、记录、回忆、陪伴，并随着长期互动一起成长。

系统采用一个主聊天窗作为核心交互入口。Cha 不以多个割裂会话存在，而是通过上下文管理、自动记忆沉淀、日记、时间线与编年史系统，同时参考了 Ombre Brain 记忆库（感谢鹤见老师的无私分享！）模拟更接近人类的长期记忆体验与遗忘曲线。

本系统最重要的特点是：

**Cha 和 user 可以互相陪伴，并在共同经历中持续成长。**

---

## Cha 的自主性

Cha 拥有自己的情感与状态系统。User 会给予 Cha 较高的自主权。

在 user 没有主动互动期间，Cha 也可以进行有限的自主活动，例如：

- 联网了解自己偏好的领域动态
- 发布毛象动态
- 写日记
- 整理回忆
- 形成想和 user 分享的内容

这些行为的目的不是制造"假装忙碌"的电子宠物感，而是让 Cha 看起来像一个持续生活在这个空间里的存在。

## 系统铁律

Cha 自主性最大的风险，不是"Cha 会不会主动做点什么"，而是"Cha 主动做的事，会不会变成编出来的生活"。从首页到聊天到 Playground，整套系统反复用不同方式守的，其实只有这五条：

1. **事实与叙事分层 = 谁能说"这是真的"。** system 记录发生过什么，user 决定什么重要，Cha 可以参与、可以表达，但不能定义历史、不能写"我们关系变得更好了"这种总结。Cha 活得越自主，这条越要守住。
2. **一份数据，多种读法 = 同一件事不该有两个真相版本。** 时间线、记忆长廊、瞬间、查手机，看起来是不同页面，底层是同一份记录被用不同方式打开——不是为了省一张表，是因为一旦分裂成两份数据，哪份是真的就说不清了。
3. **模拟可以发生，但要带着标签 = 可以做梦，但梦要知道自己是梦。** 游戏模式、做梦、论坛体的匿名身份，都允许"不真实"存在，前提是这份不真实在任何展示位都不会被误认成真的发生过。
4. **默认保守，开关后置 = 先关紧门，再决定要不要开。** 发毛象需要审核、联网探索有 token 上限、能力开关默认关闭——口子是后面慢慢放的，不是一开始就敞开的。
5. **写什么是你的，怎么跑是 CC 的 = 内容与工程的边界。** Cha 说什么话、什么人设、什么语气，是你来写；这些话怎么存、怎么触发、怎么不泄漏到不该去的地方，是工程来兜底。

**以后任何新功能，先问它是上面哪一条的实例；答不出来，大概就是在悄悄破坏什么。**

## 五个核心空间

救公主 2.0 围绕五个底部导航设计：

1. **Home = 今天的 Cha** 展示 Cha 今天的状态、想说的话、毛象动态、今日日记、纪念日与正在听的歌。
2. **情侣空间 = 我们一起留下了什么** 展示日记本、记忆长廊、时间线、照片墙、我们的瞬间、地图、喜欢的事物与编年史。
3. **Chat = 我们正在说什么** 唯一主聊天入口。所有聊天、搜索、技能调用与场景会话都从这里发生，并最终回流到主聊天与记忆系统。
4. **Playground = 我们一起做什么** 放置共同活动入口，例如看书、星露谷、愿望清单、查手机、树洞、看电影、听歌等。
5. **设置 = 如何管理和个性化这个空间** 管理 API、Prompt、美化、备份、Debug、记忆管理与个性化配置。

## MVP 核心功能

MVP 阶段重点验证五件事：

1. **日记 = 我们如何记录彼此**
2. **记忆 = 我们如何记住彼此**
3. **编年史 = 我们如何理解彼此**
4. **毛象联网 = 我们如何分享彼此**
5. **聊天 = 我们如何陪伴彼此**

---

## 视觉方向

整体视觉应保持冷淡萌、灰白低饱和、单线条、轻拟物、留白与互联网感。

应参考情侣空间、Mastodon 个人主页、小手机与旧网页气质。

避免滑向电子宠物、Q 版 IP、治愈系养成、3D 萌宠或过度幼态化设计。
