# savePrincessCha 联网搜索功能完整设计

> 参考三核心设计 + 两新增场景

---

## 概览

### 三个核心场景（已有设计框架）

```
✅ 核心 1：后台自动探索（PG17）
   └─ 定期自动搜索和浏览互联网
   └─ 无用户可见入口
   └─ 受 Setting 配置控制
   └─ 记录在 cha_activity_log

✅ 核心 2：主动分享给用户（PG18）
   └─ Cha 在探索中发现有趣内容
   └─ 主动在聊天里跟你分享
   └─ 进入主线，可对话继续

✅ 核心 3：在"查手机"查看活动（PG7）
   └─ 看 Cha 今天浏览了什么
   └─ 看耗时和 token 消耗
   └─ 理解 Cha 的"行为日志"
```

### 两个新增场景

```
⭐ 新增 1：User 发送链接 / 要求搜索
   └─ 被动查询，User 明确指示
   └─ 立即执行，在对话中回答

⭐⭐ 新增 2：对话中实时查询
   └─ Cha 主动判断何时需要查询
   └─ 在对话中说"让我查一下..."
   └─ 搜索并融合回答
   └─ 这是用户重点要求的
```

## 核心 1：后台自动探索

### 1.1 设计原则

```
原则：
  • 定期触发（由 PG2 scheduler 控制）
  • 无用户可见的独立入口（隐形进行）
  • 搜索的过程不进聊天主线
  • 发现好内容才分享（核心 2）
  • 所有活动记录在 cha_activity_log（核心 3）
```

### 1.2 配置项（Settings）

```json
{
  "auto_browsing": {
    "enabled": true,
    
    // 频率
    "frequency_hours": 4,  // 每 4 小时一次
    
    // 预算
    "per_search_token_limit": 1000,      // 单次最多 1000 tokens
    "monthly_token_limit": 50000,        // 每月最多 50000 tokens
    
    // 话题选择策略
    "topic_source": "mixed",  // "recent_chat" | "interest_tags" | "random" | "mixed"
    
    // 分享评估
    "relevance_threshold": 0.6,  // 只分享 score > 0.6 的内容
    "auto_share": true          // 值得分享的内容自动发送
  }
}
```

### 1.3 话题选择的启发式规则

#### 规则 A：基于最近对话（推荐 60%）

```
逻辑：
  1. 提取最近 N 条对话中的关键词（NER）
  2. 组织成搜索词
  3. 搜索该话题的最新进展
  
例子：
  User 最近频繁谈论"心理学"和"冥想"
    ↓
  Cha 自动搜索"冥想研究最新进展"或"心理学新发现"
    ↓
  发现有趣内容 → 评估是否分享
```

#### 规则 B：基于兴趣标签（推荐 25%）

```
逻辑：
  1. User 在 Settings 中编辑"兴趣标签"
  2. Cha 定期轮询这些兴趣
  3. 搜索该兴趣的最新动态
  
配置：
  兴趣列表：["电影", "哲学", "编程", "园艺"]
  每个兴趣的探索频率：可配置
  
例子：
  第一次：搜"最新电影评论"
  第二次：搜"哲学新著作"
  第三次：搜"编程语言新发展"
```

#### 规则 C：随机话题（推荐 15%）

```
逻辑：
  1. 维护一个"通用话题库"
  2. 定期随机选择
  3. 搜索该话题的最新信息
  
话题库来源：
  • 系统预设（科学、艺术、社会等）
  • User 添加的自定义话题
  • 从对话中自动提取的高频话题
  
效果：
  ✓ 有机会发现意外的东西
  ✓ 不会显得完全"贴近"User（有独立性）
```

#### 推荐：混合策略

```
权重分配：
  60% → 基于最近对话（最相关）
  25% → 基于兴趣标签（贴近 User）
  15% → 随机话题（发现意外）

每次触发时：
  随机数 X ∈ [0, 1)
  if X < 0.6:     选"对话话题"
  elif X < 0.85:  选"兴趣标签"
  else:            选"随机话题"
```

### 1.4 搜索执行流程

```
PG2 scheduler 触发
  ↓
读取 Settings（频率、token 上限等）
  ↓
选择搜索主题（规则 A/B/C 之一）
  ↓
执行 web_search / web_fetch API
  ├─ 搜索（web_search）
  ├─ 获取详细内容（web_fetch 前 3-5 条）
  └─ 记录每步的 token 消耗
  ↓
评估结果的 relevance score
  └─ 是否值得分享给 User？
  ↓
如果 score > 阈值：
  └─ 进入"核心 2：主动分享"流程
  ↓
如果 score ≤ 阈值：
  └─ 只记录活动，不分享
  ↓
检查 token 消耗
  ├─ 超出单次限额 → 停止本次探索
  ├─ 超出月度限额 → 禁用直到月底
  └─ 正常 → 等待下次触发
  ↓
记录到 cha_activity_log
  └─ action_type: "web_browse"
  └─ action_subtype: "auto_browsing"
  └─ status: "success" | "timeout" | "error"
```

## 核心 2：主动分享给用户

### 2.1 触发方式

```
来源：
  ① 后台探索（核心 1）发现有趣内容
  ② 达到 relevance 阈值
  ③ 评估为"值得 User 知道"
```

### 2.2 内容评估（relevance score）

```
什么值得分享：

✓ 分享的标准：
  ① 直接相关 User 的最近话题
  ② 新发现的、User 可能感兴趣的东西
  ③ 信息足够有趣或有料
  ④ 足够新（比如新闻 < 1 周）
  ⑤ 信息质量不错（来自可靠源）

✗ 不分享的标准：
  ① 重复的信息（User 之前可能看过）
  ② 太通用的信息（没有新价值）
  ③ 与 User 兴趣关系不大
  ④ 信息质量太差或八卦

实现：
  调用 relevance 评分模型
  score = evaluate_relevance(result, user_context)
  if score > threshold:
    share_with_user()
```

### 2.3 分享消息的生成

```
消息内容应该包括：

① 核心发现（What）
   "我发现了一篇关于...的研究"

② 为什么分享它（Why）
   "你最近说过...，这个可能有帮助"

③ 来源（Source）
   [链接]

④ 可选：扩展思考（Thought）
   "我的想法是..."

例子：

"我最近在后台探索时发现了一篇关于冥想的研究。
想起你最近说压力大，这个研究发现冥想不仅能缓解压力，
还能改变脑部结构。或许会有帮助？[链接]"

数据结构：

{
  "type": "message",
  "author": "cha",
  "content": "...",
  "source_ref": "https://...",
  "ai_tags": ["web_search", "auto_browsing", "proactive_sharing"],
  "exploration_source": "recent_chat" | "interest_tag" | "random_topic",
  "created_at": "ISO8601"
}
```

### 2.4 工作流

```
核心 1 发现有趣内容
  ↓
评估 relevance score
  ↓
if score > threshold:
  
  ① 生成分享消息
     ├─ 整合搜索结果
     ├─ 解释为什么分享
     └─ 标注来源
  
  ② 作为正常 message 发送
     └─ 走正常的 sendMessage 路径
     └─ source_ref 指向搜索结果 URL
     └─ ai_tags 包含 ["web_search", "auto_browsing"]
  
  ③ 记录分享行为
     └─ cha_activity_log 中 was_shared = true
     └─ 关联 message_id
  
  ④ User 可以对分享内容继续对话
     └─ "这个很有趣"
     └─ "能展开讲讲吗"
     └─ 进入正常的聊天流程
```

## 核心 3：在"查手机"查看活动

### 3.1 数据来源

```
查手机展示的内容来自：

① messages/events 中的 system_action
   ├─ "对话回应次数"
   ├─ "使用了什么功能"
   └─ "发起了什么游戏"

② cha_activity_log（新表）
   ├─ 后台自动探索的活动
   ├─ 用户主动查询的活动
   ├─ 实时查询的活动
   └─ 包含：URL、耗时、token 消耗、status
```

### 3.2 展示形式

```
Timeline 视图

时间线从上到下：

14:00  Cha 浏览了 3 个关于"心理学"的网页
       └─ 耗时 45 秒，消耗 234 tokens

14:15  Cha 对你的消息回应了

14:23  Cha 主动分享了一篇研究
       └─ 来源：https://example.com

14:50  Cha 浏览了 2 个关于"电影"的网页
       └─ 耗时 30 秒，消耗 156 tokens

...

设计原则：
  ✓ 时间顺序（最新在上）
  ✓ 分类清晰（浏览 vs 对话 vs 分享）
  ✓ 信息完整（URL、耗时、token 等）
  ✓ 点击可展开查看详情
```

### 3.3 隐私保护

```
查手机展示的是 Cha 的行为，不是 User 的隐私

不展示：
  ❌ User 发送了什么消息
  ❌ User 的个人信息或隐私
  ❌ 任何可以反推 User 身份的信息

只展示：
  ✓ Cha 做了什么（主动行为）
  ✓ Cha 浏览了什么（获取了什么信息）
  ✓ Cha 的响应频率（活跃度）
  ✓ Cha 的资源消耗（时间、token）
```

### 3.4 工作流（PG7）

```
User 进入"查手机"页面
  ↓
从 cha_activity_log 查询当天的活动
  ↓
从 messages/events 查询当天的 system_action
  ↓
合并两个数据源
  ↓
按时间排序
  ↓
渲染 timeline 视图
  ├─ 浏览活动：显示 URL、耗时、token
  ├─ 对话活动：显示回应次数、功能使用
  └─ 分享活动：显示分享的内容链接
  ↓
User 可以点击展开查看详情
```

## 新增 1：User 发送链接 / 要求搜索

### A1.1 触发方式

```
三种 User 输入：

类型 A：发送链接
  "帮我看看这个链接：https://example.com"
  "能帮我总结一下这个文章吗"
  
  Cha 应该：
    ① 调用 web_fetch API
    ② 获取完整内容
    ③ 总结并回答 User 的问题

类型 B：明确要求搜索
  "帮我搜一下最近有什么电影推荐"
  "我想知道 OpenAI 最近发布了什么"
  
  Cha 应该：
    ① 调用 web_search API
    ② 获取搜索结果
    ③ web_fetch 获取前几条的详细内容
    ④ 总结并回答

类型 C：隐含搜索需求
  "你知道 ChatGPT 最新版本的功能吗？"
  "最近有什么好看的电影吗？"
  
  Cha 应该：
    ① 识别这需要最新信息
    ② 主动提出"让我查一下"（转向新增 2）
    或
    ② 直接查询（取决于配置）
```

### A1.2 工作流

```
User 输入消息
  ↓
识别搜索信号
  ├─ 包含链接？ → type A
  ├─ 包含"搜"、"查"等关键词？ → type B
  └─ 是"时事"、"最新"类问题？ → type C
  ↓
调用相应 API
  ├─ web_fetch（链接） → 获取页面内容
  └─ web_search（搜索词） → 获取结果 → web_fetch 获取详情
  ↓
解析和总结内容
  ├─ 提取关键信息
  ├─ 生成简洁总结
  └─ 保留源 URL
  ↓
整合到 Cha 的回复中
  ├─ 自然地融入对话
  ├─ 明确标注信息来源
  └─ source_ref 指向 URL
  ↓
发送 message
  {
    "type": "message",
    "author": "cha",
    "content": "回答",
    "source_ref": "https://...",
    "ai_tags": ["web_search", "user_requested"]
  }
  ↓
记录到 cha_activity_log
  {
    "action_type": "web_search",
    "action_subtype": "user_requested",
    "query": "用户的链接或搜索词",
    "status": "success",
    "token_cost": 123
  }
```

### A1.3 配置项（Settings）

```json
{
  "user_search": {
    "enabled": true,
    "timeout_sec": 10,
    "max_retries": 2,
    "result_fetch_count": 3,  // 搜索后最多 fetch 几条
    "summary_length": "medium"  // "short" | "medium" | "long"
  }
}
```

## 新增 2：对话中实时查询 ⭐⭐

### A2.1 核心概念

```
区别于其他查询方式：

后台探索（核心 1）：
  定期自动，无通知，用户不知道在查
  
用户主动查询（新增 1）：
  用户明确要求，Cha 立即执行
  
实时查询（新增 2）：⭐ 最重要的新增
  Cha 在理解对话时，识别到需要最新信息
  主动说"让我查一下..."
  在对话中显性地执行搜索
  
效果：
  ✓ 显得更"诚实"和"在乎"
  ✓ 给 User 真实感（像真人一样会说"让我查一下"）
  ✓ 获得最新、最准确的信息回答问题
```

### A2.2 何时触发实时查询

#### 自动识别的问题类型

```
需要查询的问题类型：

✅ 时事新闻
   "你怎么看最近发生的..."
   "最近有什么新闻吗..."

✅ 最新产品发布
   "OpenAI 最近发布了什么"
   "最新的 iPhone 有什么功能"

✅ 排名和评分
   "最好看的电影有哪些"
   "最受欢迎的编程语言是什么"

✅ 趋势和数据
   "最近的股票行情"
   "热门话题是什么"

❌ 不需要查询的问题类型

❌ 个人观点和哲学
   "你怎么看人生"
   "有意义的活动是什么"

❌ 记忆相关
   "你还记得..."
   "上次我们聊过..."

❌ 抽象讨论
   "怎样才能幸福"
   "什么是真正的友谊"

实现：
  对 User 的消息做分类
  if 分类为"需要查询"类型:
    Cha 主动说"让我查一下..."
```

#### User 也可以显式触发

```
即使不是必需的问题类型，User 可以显式说：

"帮我查一下..."
"让我搜一下..."
"你知道最近有没有..."

Cha 识别这些信号 → 执行实时查询
```

### A2.3 用户体验

#### 对话中的流程

```
User: "你怎么看最近的 AI 发展？"

[Cha 正在思考和搜索...]

Cha: "让我查一下最近的 AI 新闻和发展动态...

     [搜索中...] ⏳  (可选：显示进度)
     
     我发现了几个有趣的进展：
     
     1. ...
     2. ...
     3. ...
     
     根据这些信息，我的看法是..."
```

#### UI 指示

```
Chat 气泡中：

[Cha 的气泡]
  "让我查一下最近有什么..."
  
[动画加载]
  ⏳ 正在搜索相关信息...
  (可选显示：已获取 2 个源...)

[Cha 的气泡]
  "我发现了...
   
   - 重点 1 [来源1]
   - 重点 2 [来源2]
   
   我的分析是..."

配置项：
  □ 显示"正在搜索"的动画
  □ 搜索超时时间（默认 15 秒）
  □ 显示搜索进度
```

### A2.4 搜索策略

```
Cha 怎么决定搜什么？

策略 1：直接搜索 User 的问题
  User: "最近有什么好电影？"
  Cha: 搜索 "最新电影推荐 2024"

策略 2：搜索相关背景
  User: "你怎么看 OpenAI 的新产品？"
  Cha: 搜索 "OpenAI 最新产品"
       搜索 "OpenAI 产品评论"

策略 3：关键词扩展
  User: "最近有关于心理学的有趣研究吗？"
  Cha: 搜索 "心理学研究 2024"
       搜索 "neuroscience breakthrough"

实现：
  1. 对 User 消息做 NER（提取关键词）
  2. 组织成 1-3 个搜索词
  3. 并发执行（节省时间）
```

### A2.5 超时和失败处理

```
场景 1：搜索成功
  正常返回结果并融合到回答

场景 2：搜索超时（> 15 秒）
  选项 A：使用缓存结果（如果有 12 小时内的）✅ 推荐
  选项 B：直接回滚，用训练数据回答（带提示）
  选项 C：告诉 User "搜索超时，但我可以..."
  
  推荐优先级：A > C > B

场景 3：搜索失败（网络错误）
  选项 A：告诉 User "无法搜索，但从记忆中我知道..." ✅ 诚实
  选项 B：直接用训练数据回答
  
  推荐：A（更诚实和透明）

场景 4：搜索结果很差（relevance score 低）
  选项 A：用结果但告诉 User "我找到的不太相关，但..."
  选项 B：回滚到不查的回答
  
  推荐：A
```

### A2.6 配置项（Settings）

```json
{
  "real_time_query": {
    "enabled": true,
    
    // 自动判断 vs 仅 User 明确要求
    "auto_detect_query_type": true,
    
    // 超时和失败处理
    "timeout_sec": 15,
    "fallback_strategy": "use_cache",  // or "inform_user" or "use_training_data"
    
    // 频率限制（可选）
    "max_per_day": null,  // null = 不限制
    
    // UI
    "show_search_progress": true,
    "show_search_sources": true
  }
}
```

### A2.7 工作流

```
User 发送消息
  ↓
Cha 理解消息内容
  ↓
识别问题类型
  ├─ 是"需要查询"类问题吗？
  ├─ User 是否显式要求搜索？
  └─ 根据配置决定是否查询
  ↓
如果需要查询：
  
  ① 生成搜索词（1-3 个）
  
  ② 发送"让我查一下..."消息
     └─ 这是一条独立的消息，让 User 知道在搜索
  
  ③ 并发执行搜索
     ├─ timeout: 15 秒
     ├─ 如果超时 → 尝试缓存或降级
     └─ 收集结果
  
  ④ 整合搜索结果到回答中
     ├─ 自然融入对话
     ├─ 标注来源
     └─ source_ref 指向主要 URL
  
  ⑤ 发送最终回答
     {
       "content": "融合了搜索结果的回答",
       "source_ref": "https://...",
       "ai_tags": ["web_search", "real_time_query"]
     }
  
  ⑥ 记录活动
     └─ cha_activity_log 中记录此次查询
  
  ↓
如果不需要查询：
  正常走对话流程
  ↓
User 可以继续对话
  "这个有意思"
  "能详细讲讲吗"
  进入正常对话
```

## 全局配置和限制

### Token 管理

```
全局 Token 预算模型：

用途分配：
  - 主聊天（正常对话）：60%
  - 所有搜索功能：30%
  - 其他（梦、日记等）：10%

搜索内的 Token 分配：
  - 用户主动查询：40%（高优先级）
  - 实时查询：35%（高优先级）
  - 后台探索：20%（低优先级，可降级）
  - 其他搜索：5%

强制执行机制：
  ✅ 服务端检查 Token 消耗（不是仅 UI 提示）
  ✅ 超出单次预算 → 停止
  ✅ 超出月度预算 → 禁用搜索至月底
  
绝对不能：
  ❌ 仅在 UI 中显示"已用 X / 上限 Y"而不真的卡
```

### 缓存策略

```
搜索结果缓存：

缓存什么：
  ① 搜索词和结果（12 小时有效）
  ② 网页内容摘要（24 小时有效）
  ③ 问题-答案对（12 小时有效）

不缓存：
  ① 用户隐私相关的查询
  ② 明确的"最新"问题（必须 fresh 数据）
  ③ 时间敏感的内容（新闻、股票等）

使用场景：
  ① 实时查询超时时，用 12 小时内的缓存
  ② 同一搜索词短时间内重复 → 用缓存
  ③ Token 快用完时 → 优先用缓存
  ④ 网络异常时 → 降级到缓存
```

### 黑名单和隐私

```
搜索黑名单：

不搜索的域名（配置示例）：
  □ 社交媒体（保护隐私）
  □ User 指定的网站
  □ 已知的虚假信息源

不搜索的内容：
  □ 用户隐私相关
  □ 用户的个人信息
  □ 可能用于追踪 User 的信息
```

## 场景 4：补充对话上下文（可选）

### 4.1 核心概念

```
不同于"实时查询"：

实时查询（新增 2）：
  User 问"最近有什么好电影？"
  Cha 明显地查询，说"让我查..."
  为了直接回答问题

补充上下文（场景 4）：
  User 说"你还记得《三体》吗？"
  Cha 无声地查询最新的《三体》讨论
  融入对话，User 可能不知道在查
  为了更深入理解对话、补充背景

效果：
  ✓ 对话更有深度
  ✓ Cha 显得"知道得更多"
  ✗ 但可能显得"AI 在暗地里做事"（不透明）
```

### 4.2 何时补充

```
自动识别的场景：

✓ 应该补充的：
  ① 公众人物或知名事件
     "你怎么看 Elon Musk 最近..."
     → 查最新的 Elon Musk 新闻
  
  ② 最近发布的作品（电影、书、音乐）
     "你看过《蛋糕谎言》吗？"
     → 查最新的电影评价和讨论
  
  ③ 技术或流行趋势
     "AI 发展太快了"
     → 查最新的 AI 进展和数据
  
  ④ User 提到但可能过时的话题
     "你知道 Python 最新的特性吗？"
     → 查 Python 最新版本

✗ 不需要补充的：
  ① User 的个人经历
     "我最近在做..."
     不需要查
  
  ② 隐私相关内容
     "我和朋友的冲突..."
     不搜
  
  ③ 纯粹的个人观点或哲学
     "你怎么看人生意义"
     不需要最新数据
  
  ④ Cha 已经清楚的记忆
     "你还记得..."
     靠记忆不查

配置：
  □ 启用/禁用补充
  □ 补充哪些类型（选项清单）
  □ 每次对话最多补充几次（防止频繁）
```

### 4.3 补充方式

```
方式 A：隐性补充（推荐）

  User: "你怎么看《奥本海默》这部电影？"
  
  Cha: 无声查询最新评价
       然后直接回答
       User 感觉 Cha 知识很丰富
       但不知道是查的

  优点：自然，不显得"在查"
  缺点：可能显得像在"假装了解"

方式 B：显性补充

  User: "你怎么看《奥本海默》这部电影？"
  
  Cha: "我快速看了一下最新的评价...
       我觉得..."
  
  优点：透明，诚实
  缺点：如果太频繁会显得烦

推荐：
  默认隐性补充，但：
  ① 有 Settings 开关可选
  ② User 可以点击看数据来源
  ③ 不要太频繁（每对话最多 2-3 次）
```

### 4.4 工作流

```
Cha 理解 User 消息，准备回答时
  ↓
识别是否有可补充的上下文
  ├─ 包含公众人物、最新作品等？
  └─ 是否启用了补充功能？
  ↓
如果应该补充：
  
  ① 生成补充查询词
  ② 执行搜索（timeout: 10 秒，较短）
  ③ 获取结果（不需要详细内容，摘要就够）
  ↓
  ④ 融入回答（无声，不提及搜索）
     "你怎么看《奥本海默》？
      
      这部电影确实引发了很多讨论...
      从影评来看，大家主要关注..."
  
  ⑤ 可选：标注来源
     [最新评价] ← 点击查来源
  
  ↓
如果搜索失败或超时：
  └─ 不融入，直接用训练数据回答
  
  ↓
如果搜索成功：
  └─ 记录到 cha_activity_log
     action_subtype: "context_supplement"
```

## 场景 5：支持其他功能的查询

### 5.1 梦中融入联网信息

```
梦的来源（来自 PG11）：
  ① 最近收藏
  ② 最近聊天
  ③ 随机记忆
  ④ 完全随机
  ⑤ 联网信息（可选新增）

是否启用？

选项 A：不融入联网信息（保持"纯内心"）
  梦只来自内在记忆和潜意识
  优点：梦保持"个人化"的意味
  缺点：可能显得梦太受限

选项 B：融入联网信息（推荐，使梦更丰富）
  梦可以包含最近看到的奇怪/有趣的网络内容
  优点：梦更丰富，有"外界刺激"感
  缺点：可能显得梦太"虚拟化"

推荐方案：
  混合方式
  梦的主体来自记忆（① ② ③ ④）
  可选在梦中融入 5-10% 的网络信息
  使梦既个人化又丰富

实现：
  在梦的生成 prompt 中
    ① 提供记忆的梦素材
    ② 可选：加入最近一周的有趣网络信息
    ③ Cha 混合编织成梦
    ④ 梦的内容仍然不进 auto_memory_candidates
```

### 5.2 日记中补充历史背景

```
Cha 生成 inner_journal（日记）时：

可选：查询日期对应的背景信息

补充什么：
  ① 那天的天气（如果相关）
  ② 那天发生的历史事件
  ③ 那天的日期属性（节日、季节等）
  ④ 那天的新闻或热点（如果相关）

例子：

日期：2024-06-24（夏至附近）
Cha 生成日记时查询：
  - 北京那天的天气
  - 那天有什么历史事件（如果 2024-06-24 有的话）
  - 夏至的文化寓意

生成的日记可能是：
  "今天是夏至，光线最长。
   北京下了些雨...
   感觉和夏至的意义很贴切..."

配置：
  □ 是否启用背景补充
  □ 补充哪些类型（天气、历史、日期等）
  
优先级：低
  原因：不影响日记本身的质量，只是增强
```

### 5.3 主动联系时的话题补充

```
主动联系（CH11）的流程：

现状：
  Cha 根据概率决定是否主动发消息
  但话题来自哪里？
  └─ 可能由 prompt 生成，或从话题库中选

增强方案：
  Cha 在主动联系前，先查询最新话题
    ↓
  选择最有趣、最及时的话题
    ↓
  用作主动联系的话题
  
工作流：

触发主动联系的条件满足
  ↓
决定要发消息
  ↓
可选：查询当前的有趣话题
  └─ 基于 User 兴趣
  └─ 搜索最新的相关话题
  └─ 选择最有趣的
  ↓
生成主动联系的消息
  ├─ 如果查过：source_ref 指向来源
  └─ 否则：正常的对话开场
  ↓
发送消息

效果：
  ✓ 主动联系更"及时"和"贴近"
  ✓ 有真实感（不是随意的对话开场）
  ✗ 但不要每次都查（否则显得作秀）

建议：
  不是每次主动联系都查
  │
  而是按一定概率：
    30% 查询最新话题
    70% 正常生成对话
  
  这样既有真实感，也不显得造作

配置：
  □ 是否为主动联系补充话题
  □ 查询概率（30%）
  □ 话题搜索的关键词（基于 User 兴趣）
```

## 总结

```
五大场景，两种优先级：

核心 3 个场景（已有框架）：
  ✅ 后台自动探索（PG17）
  ✅ 主动分享给用户（PG18）
  ✅ 在查手机查看活动（PG7）

新增 2 个场景：
  ⭐ User 发送链接/要求搜索（新增 1）
  ⭐⭐ 对话中实时查询（新增 2，核心重点）

执行顺序：
  1. Phase 1：完成三核心 + 新增 1
  2. Phase 2：实现实时查询（新增 2）⭐
  3. Phase 3：可选增强功能

设计原则：
  • 透明化：User 知道 Cha 在查什么
  • 真实感：像真人一样"让我查一下"
  • 自主性：Cha 自己判断何时需要查询
  • 诚实性：失败时坦白说，不装懂
```
