# savePrincessCha 2.0 — 架构总览

> 这份文档给你一张全景图：五个 PRD 之间怎么关联、哪些 task 是"地基"必须先做、哪些能并行、哪些卡在内容资产等不了。不是让你把所有 PRD 读一遍，而是让你知道"现在该做什么、下一步能做什么"。

---

## 系统地基三层（一切都卡在这些上）

```
🟢 第 0 层：数据表（最底层）
├─ messages → events 单表（扩展现有，+5 字段）
│  └─ Task: CH1（聊天页）= S1（情侣空间）【做一次，两页共用】
├─ cha_activity_log → 行为日志（查手机 + 联网探索）
│  └─ Task: PG1（Playground）
└─ app_settings → 配置中心（控制整个系统）
   └─ Task: ST1（设置页）

⏱ 预计：CH1 + PG1 + ST1 可以并行，各 1-2 天
🔴 红线：三张表的 schema 必须在第一周内定死，后面所有东西都建在这上面
```

## 优先级队列（建议执行顺序）

### 🔴 第一周：地基零期（必须）
```
同时开工：
□ CH1  - messages 表扩展
□ PG1  - cha_activity_log 表  
□ ST1  - app_settings 表

预计耗时：1-2 天（三个并行）
关键度：最高，一个错了全部要返工
```

### 🟠 第二周：横切 + 调度
```
完成顺序：
□ C1   - 编辑态组件
□ C2   - 上传组件
□ PG2  - scheduler 框架

预计耗时：2-3 天
关键度：高，后续 job 都卡在 PG2
```

### 🟡 第三周+：业务模块
```
推荐优先级（按 MVP 五件事排）：

1️⃣  聊天核心（CH2-CH6） ← 验证记忆沉淀是否生效
2️⃣  inner_journal（D1-D6） ← 验证日记和记忆联动
3️⃣  毛象（B1-B2） ← 最轻，快速验证数据流
4️⃣  记忆系统（S2/T1-M2） ← 用真实数据验证多视图
5️⃣  其他（游戏/Playground/朋友圈）

建议分批发任务，不要一次扔 20 个 task。
```

## 红线清单（任何 task 都不能违反）

### 🔴 代码层红线
```
□ 永不生成 self_narrative（关系叙事）
  - 日记里不写"我们变得…"
  - 串词里不写"我感觉你…"
  - 记忆总结不写第一人称观点
  
□ 一份数据，多种视图（不分裂存储）
  - 时间线、长廊、瞬间都来自同一张表
  - 不能有两个"真相版本"
  
□ 模拟和真实明确分开（tag + 隔离）
  - 游戏对话 = sandbox，不进 events
  - 做梦 type=dream，time_line 显式排除
  - 论坛匿名内容独立表存储
  
□ cha 能行动，不能定义事实
  - cha 可以说话、表达、参与
  - 但不能写"这是真实发生过的"
  - 权重决策权只有 user（是否收藏）和 system（是否记录）
  
□ 默认保守，开关后置
  - 朋友圈/毛象先审核再发
  - 联网探索有 token 上限（服务端强制）
  - 功能开关默认关闭
```

### 🟡 架构层红线
```
□ CH1 = S1 做一次
  - 不能分成两个 task
  - messages 表的扩展一次性完成
  
□ PG2 是 CH11/PG17/PG14 的必需前置
  - scheduler 框架必须先建
  - 后续所有定时 job 都挂在这上面
  
□ B1 API 层复用到 F2（发毛象）
  - 不能另起 Mastodon 调用链
  - 发和读共用同一套 API 封装
  
□ 两张表不能合并（PG1 ≠ CH1）
  - cha_activity_log 和 messages 是平行的两条线
  - 分别记"说了什么"和"自己做了什么"
```

## 跨 PRD 依赖关系（最容易踩的坑）

```
聊天页 CH11 ← Playground PG2
  问题：CH11（主动联系触发）需要 scheduler 框架
  后果：PG2 没做完，CH11 无法开工
  
首页 E1-E4 ← Playground PG2
  问题：音乐的联网探索需要 scheduler
  后果：音乐系统要等 PG2
  
首页 B1 ← Playground F2
  问题：发毛象的 API 层复用首页拉取的代码
  后果：B1 必须先于 F2，或 B1 API 封装要提前独立出来

Playground PG12-16（做梦）← 聊天页"dream"type
  问题：做梦的 messages 记录要用 type=dream
  后证：CH1 的 type 枚举里必须预留 dream 值
  
所有依赖 events 的 ← 聊天页 CH1
  问题：CH1 是五个 PRD 的共同地基
  后果：CH1 一改，所有读 messages 的地方都要适配
```

## 红线总结（一句话版）

| 红线 | 一句话 | 如果违反 |
|------|--------|---------|
| 不生成 self_narrative | 记事实，不讲故事 | 用户分不清真实和虚构 |
| 一份数据多视图 | 同一件事不该有两个版本 | 历史对不上，修复困难 |
| 模拟带标签 | 做梦的内容要标记成梦 | 用户把虚构当成真的 |
| 不定义事实 | cha 参与不定义 | 关系叙事变成伪装生长 |
| 默认保守 | 先关门，后放开 | 无法撤回的不良后果 |

## 接下来该干什么

**如果你刚拿到这个文档：**
1. 先读 Vision.md（5 分钟）
2. 再看本文的"任务执行依赖树"（5 分钟）

1. 来这份文档查"跨 PRD 依赖关系"
