# 记忆系统设计实战反馈 Memory Design Feedback

> 状态：已定稿
> 来源：Phase 3–4 实战调试积累
> 本文件记录记忆系统在真实使用中暴露的问题、决策过程和教训，供后续迭代参考。

---

## 一、已解决的问题

### 1. story_seeds 盲注导致 AI 复述硬编码叙事

**现象**：AI 在回复中主动引用"第一次说爱"等写死故事，脱离当前对话语境。
**根因**：story_seeds 是基于主题词的全量注入，触发阈值过低（任何情感词都可能命中）。
**解决**：完全停用 story_seeds。关系叙事改由用户主动在对话中提及，不再通过系统强制注入。
**教训**：硬编码叙事本质上是在替用户"记住"她应该有什么感受，违反了"记照护方式"原则。

---

### 2. memory_buckets 无关键词时盲注

**现象**：没有 keywords 字段的 bucket 条目通过 title+summary token 弱匹配注入，噪声高。
**根因**：弱匹配规则过于宽松，几乎所有对话都会命中若干 bucket。
**解决**：LEGACY_MEMORY_ENABLED 默认关闭，legacy 路径整体停用。
**教训**：宽松的关键词匹配比没有匹配更危险，因为它给人"有过滤"的错觉。

---

### 3. mastodon_profile 误被要求"自我介绍"

**现象**：AI 把 profile 内容当成可对外复述的信息，主动说"你的 profile 里写了……"。
**根因**：系统提示没有明确区分"读取理解"和"对外复述"。
**解决**：profile 注入包装为 `<user_core_profile>`，系统提示中说明"参考资料，不用于复述来源"。
**教训**：数据的包装方式（XML 标签名称）���模型的理解方式有实质影响。

---

### 4. timeline 常驻导致 token 浪费

**现象**：早期版本 timeline 每轮注入，在简单日常对话中占用大量 context。
**根因**：timeline 文件体积大（完整时间线 markdown），不适合常驻。
**解决**：改为 retrieval_only，由 `shouldInjectTimeline()` 按需判断。
**教训**：token 成本不只是钱的问题，更是注意力的竞争：无关内容会稀释有用上下文。

---

### 5. 时间类问题答错（猜测日期）

**现象**：用户问"我们认识多久了"，AI 给出了一个看起来合理但不准确的数字。
**根因**：timeline 没有注入，但模型基于 profile 的碎片信息拼凑了一个答案。
**解决**：
1. 系统提示中加入"涉及时间事实问题，记忆中无明确记录时只说'不清楚'"。
2. timeline 触发词增加了"纪念日"、"第几天"等关键词。
**教训**：模型宁可给出自信的错误答案，也不愿说"不知道"。必须在系统提示层面强制约束。

---

## 二、设计决策记录

### 决策 A：profile 用 TS 模块内联而非文件读取

**背景**：Supabase Edge Functions 的 Deno 运行时无法可靠读取同目录文件（部署路径问题）。
**决策**：把 profile.md 和 timeline.md 编译为 TS 模块的 `export const` 字符串常量。
**代价**：每次更新 profile/timeline 内容需要重新部署函数。
**可接受性**：profile 是低频更新的静态内容，月更一次，代价可接受。

---

### 决策 B：不使用向量数据库

**背景**：用向量相似度召回可以更精确地匹配语义相关记忆。
**决策**：Phase 4 不引入向量库，继续使用关键词触发。
**原因**：
- 向量库引入了外部依赖和延迟
- 对于已知的有限场景（年份/地点/里程碑），关键词足够精确
- 向量召回���"相似度"不等于"应该告诉用户"，隐私边界更难控制
**未来**：ombre_vault 可能引入轻量向量索引，但需要先解决权限和删除问题。

---

### 决策 C：Memory Center 第一版只展示状态，不做编辑

**背景**：展示层和编辑层的复杂度差异巨大。
**决策**：第一版只读，状态可见。编辑（禁用/删除 provider）留到第二版。
**原因**："可见即可控"的第一步是先让用户看见，再给控制权。盲目添加控件会制造虚假的"控制感"。

---

### 决策 D：不把 profile 内容暴露给前端

**背景**：Memory Center 展示 provider 状态时，是否显示 profile 内容？
**决策**：不显示 profile 全文。只显示字符数和加载状态。
**原因**：
- profile 包含敏感信息（诊断、家庭创伤、人际关系）
- 前端是面向用户的界面，这些信息本应由用户自己知晓，不需要系统"展示回来"
- 展示的目的是让用户知道系统用了什么，而不是展示系统知道什么

## 三、待观察的问题

### P1：timeline 触发词覆盖不足

**现象**：用户问"我在澳门过得怎么样"，但"澳门"已在触发词列表中，实际测试偶尔漏召回。
**猜测**：`shouldInjectTimeline()` 逻辑正确，但 timeline.md 内容可能不包含用户期待的那段记忆。
**待做**：增加 timeline 召回的 debug 日志（`timeline_reason`），帮助定位是触发层还是内容层的问题。

---

### P2：profile 更新滞后

**现象**：用户经历了新事件（如猫咪住院），但 profile 尚未更新，AI 对此无感知。
**现状**：profile 是手动维护的静态文件，没有自动更新机制。
**待做**：Phase 5 考虑给 profile 增加"最近更新时间"标注，让 AI 在回应新事件时知道自己的知识截止点。

---

### P3：memory distill 工作流与新架构的关系

**现象**：distill 功能（从对话中提取记忆）写入 `memories` 表，但 memories 注入已停用。
**当前状态**：distill 生成的内容没有任何出口，用户看不到也用不上。
**待做**：明确 distill 的定位——是写入 ombre_vault 的前身，还是保留为独立的手动记忆库？

## 四、北极星原则的实战含义

### "默认遗忘"

不是说 AI 不记事，而是说：**没有显式存储决策，就不记**。profile 是用户自己整理的，timeline 是从公开发帖提炼的，都有明确的来源和更新流程。系统不应自动从对话中抓取内容写入任何持久层（ombre_vault 上线前）。

### "记照护方式"

不是说不记创伤，而是说：**记"怎么陪伴"，不记"发生了什么**。profile 里有"双相 II 型、8 月情绪高峰"，这是照护信息。但具体的某次崩溃、某段自责的话，不应该被系统记录和复现——那是用户选择遗忘的权利。

### "可见即可控"

不是说给用户一个能改所有东西的后台，而是说：**用户能看到系统用了什么**。Memory Center 的四张卡片就是这个原则的第一版实现。第二版会加上"禁用"和"清除"按钮。
